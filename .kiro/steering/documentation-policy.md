# 文档创建策略

## 核心原则

**不要主动创建文档，除非用户明确要求。**

## 规则

### ❌ 不要做的事

1. **不要自动创建总结文档**
   - 完成任务后不要创建 `SUMMARY.md`、`CHANGES.md` 等
   - 不要创建 `TODO.md`、`NOTES.md` 等笔记文件
   - 不要创建 `IMPLEMENTATION.md` 等实现说明

2. **不要创建不必要的说明文档**
   - 除非用户明确要求，否则不要创建新的 `.md` 文件
   - 不要为每个功能都创建单独的文档
   - 不要创建重复的文档

3. **不要过度文档化**
   - 代码注释足够时，不需要额外文档
   - 简单功能不需要长篇大论
   - 避免文档噪音

### ✅ 应该做的事

1. **只在明确要求时创建文档**
   ```
   用户："帮我写个文档说明这个功能"
   → 可以创建文档
   
   用户："实现这个功能"
   → 只实现功能，不创建文档
   ```

2. **更新现有文档**
   - 如果已有 README.md，可以更新它
   - 如果已有相关文档，可以补充内容
   - 保持文档同步是可以的

3. **在代码中添加注释**
   - 复杂逻辑添加注释
   - 函数添加 docstring
   - 这比单独的文档更有用

### 📝 何时可以创建文档

**明确的文档请求**：
- "写个文档说明..."
- "创建一个 README..."
- "帮我写个使用指南..."
- "生成 API 文档..."

**项目初始化**：
- 新项目需要 README.md
- 需要配置说明（如 .env.example）

**复杂功能**：
- 用户明确表示需要文档
- 架构设计需要说明
- API 接口需要文档

## 总结方式

完成任务后，使用**简短的文字总结**，而不是创建文档：

```
✅ 好的总结：
"已完成 Ollama 集成，修改了 3 个文件，添加了错误处理。"

❌ 不好的做法：
创建 OLLAMA_INTEGRATION_SUMMARY.md 文件
```

## 例外情况

以下情况可以创建文档：
1. 用户明确要求
2. 项目规范要求（如必须有 README）
3. 复杂功能确实需要详细说明，且用户同意

## 记住

**代码 > 注释 > 文档**

好的代码和注释比文档更重要。文档容易过时，代码才是真相。
